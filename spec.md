# Fudagit Web 仕様書

## 1. 概要

`Fudagit Web` は、Git コマンドをカルタ形式で学べる SolidJS 製のブラウザゲームである。  
プレイヤーは読み札として表示される「Git コマンドの説明文」を読み、複数の取り札の中から正しいコマンドを選択する。

このプロジェクトで最も重視する体験は、読み札の説明文が音声読み上げの代わりとして、少しずつ浮かび上がる演出である。  
単なるテキスト表示ではなく、百人一首の「読まれる間」を感じさせる UI/UX を主軸にする。

## 2. MVP ゴール

MVP では以下を実現する。

- 難易度 `easy / normal / hard` を選択して遊べる
- ゲーム開始時に問題数を指定できる
- 各問題で説明文が時間経過でじわじわ浮かび上がる
- プレイヤーが取り札をクリックして回答できる
- 回答直後に正誤、正しいコマンド、解説を表示する
- 全問終了後にスコアを表示し、再プレイできる
- PC とモバイルの両方で破綻しないレイアウトにする
- GitHub Actions で lint / format / build / GitHub Pages deploy を自動化する

MVP では以下は対象外とする。

- 音声読み上げ
- ユーザー認証
- スコア保存
- タイムアタック
- 多言語対応

## 3. 技術方針

- フレームワーク: `SolidJS`
- ビルド基盤: `Vite`
- 言語: `TypeScript`
- スタイル: `plain CSS`
- デプロイ先: `GitHub Pages`
- リポジトリ想定: `git@github.com:torifo/fudagit-web.git`

`plain CSS` を採用する理由は以下。

- 縦書き、和紙風テクスチャ、ぼかしと透明度の複合演出を素直に書きやすい
- Tailwind でも可能だが、今回の演出中心の UI は専用 CSS の方が保守しやすい
- アニメーションの意図をコンポーネントごとに明示しやすい

## 4. ゲームコンセプト

### 4.1 テーマ

- 百人一首をモチーフにした和風 UI
- Git 学習用途だが、初心者専用には寄せすぎず、一般的な開発者も楽しめる問題範囲にする
- コマンド暗記ではなく、説明文から用途を判断する体験を重視する

### 4.2 コア体験

このゲームの中心は「説明文が徐々に立ち現れる」ことである。  
視覚的には以下を組み合わせる。

- 初期状態では説明文を薄く表示する
- 初期状態では軽い `blur` をかける
- 数秒かけて `opacity` を上げる
- 数秒かけて `filter: blur()` を減らす
- 最終的に読み札がはっきり読める状態になる

プレイヤーは文が見え切る前から推測して札を取ってもよい。  
これにより、カルタらしい先読みの感覚を作る。

## 5. 難易度設計

難易度は 3 段階とする。

| 難易度 | 取り札枚数 | 想定プレイヤー           |
| ------ | ---------- | ------------------------ |
| easy   | 4 枚       | Git 初学者               |
| normal | 6 枚       | 日常的に Git を使う人    |
| hard   | 8 枚       | コマンド理解を試したい人 |

難易度によって変わるのは、MVP では取り札枚数のみとする。  
出題文の難易度調整や制限時間追加は将来拡張とする。

## 6. 問題数

ゲーム開始前に問題数を選択できるようにする。

- `5問`
- `10問`
- `20問`

出題プール数が不足する場合は、選択可能な問題数を自動調整するか、UI 上で選択肢を無効化する。

## 7. 画面構成

アプリは 3 画面構成とする。

### 7.1 スタート画面

表示要素:

- タイトル
- サブコピー
- 難易度選択
- 問題数選択
- ゲーム開始ボタン

役割:

- プレイ条件を決める
- 和風の世界観を最初に提示する

### 7.2 プレイ画面

表示要素:

- 戦績表示
- 読み札エリア
- 取り札エリア
- 回答後の結果表示エリア
- 次の問題へボタン

役割:

- 現在の問題文を読む
- 取り札を選ぶ
- 正誤と解説を確認する

### 7.3 リザルト画面

表示要素:

- 正解数
- 正答率
- プレイ条件の振り返り
- 再プレイボタン
- 設定を変えて再開する導線

役割:

- プレイ結果を振り返る
- もう一度遊ぶ導線を提供する

## 8. プレイフロー

1. プレイヤーがスタート画面で難易度と問題数を選ぶ
2. ゲーム開始時に出題プールから問題をランダム抽出する
3. 各問題で説明文を読み札として表示する
4. 説明文は数秒かけて浮かび上がる
5. プレイヤーは任意のタイミングで取り札を 1 枚クリックする
6. 回答確定後、正誤と正解コマンド、解説を表示する
7. プレイヤーが `次の問題へ` を押す
8. 最後の問題が終わったらリザルト画面へ遷移する

## 9. UI/UX 詳細

### 9.1 デザイン方針

- 百人一首をイメージした縦長レイアウトを基調にする
- 和風配色を採用する
- 和紙のような質感を背景で表現する
- ただし可読性を犠牲にしない

推奨カラー例:

- 縹色系の青
- 茜色系の赤
- 生成り系の背景
- 墨色系の本文色

### 9.2 縦書き方針

縦書きは読み札エリアを中心に採用する。  
取り札は可読性と一覧性を優先し、MVP では横書きとする。

理由:

- コマンドは横書きの方が視認しやすい
- モバイル時に縦書きの取り札はタップしづらい
- 読み札だけを縦書きにする方が、世界観と実用性のバランスがよい

### 9.3 モバイル対応

モバイルでは以下を行う。

- 1 カラム中心の縦積みレイアウトにする
- 取り札グリッドは画面幅に応じて 2 列または 1 列に調整する
- 読み札の縦書きサイズを縮小する
- タップ領域を十分に確保する
- 戦績表示はコンパクトな横並びまたは上部固定にする

### 9.4 戦績表示

プレイ中に表示する情報は以下。

- 現在の問題番号
- 総問題数
- 現在の正解数
- 正答率

## 10. 読み札アニメーション仕様

この仕様は本プロジェクトの最重要項目である。

### 10.1 演出要件

説明文は表示直後から完全に読めてはいけない。  
以下の状態遷移を持つ。

- 初期: 低い `opacity`、やや強い `blur`
- 中間: 徐々に文字が見え始める
- 終了: 通常の可読状態に到達する

### 10.2 実装イメージ

- CSS `transition` または `keyframes` を使用する
- `opacity`
- `filter: blur()`
- 必要であれば `transform` の微細な変化を加える

### 10.3 時間設計

推奨値:

- 浮かび上がり時間: `2.5秒〜4秒`

MVP では一律の演出時間でよい。  
将来的には難易度ごとに演出時間を変える余地を残す。

### 10.4 UX 上の注意

- アニメーション中でも回答可能にする
- ただし文字が完全に読めなくても、最低限の視認性は確保する
- `prefers-reduced-motion` 環境ではアニメーションを弱めるか無効化する

## 11. 回答と結果表示

### 11.1 回答方法

- 取り札をクリックまたはタップして回答する
- 1 問につき回答は 1 回のみ
- 回答後は札を再選択できない

### 11.2 回答後の表示

毎回の回答後に以下を表示する。

- 正解 / 不正解
- 正しい Git コマンド
- コマンドの解説

### 11.3 遷移方法

回答後は自動遷移させず、`次の問題へ` ボタンで進める。  
理由は以下。

- 解説を確実に読める
- 学習体験を優先できる
- モバイルでの認知負荷を下げられる

## 12. 出題データ仕様

### 12.1 データ形式

問題データは JSON で保持する。  
各レコードは以下の形とする。

```json
{
  "id": "git-status",
  "command": "git status",
  "description": "作業ツリーやステージングの状態を確認したいときに使うコマンド",
  "explanation": "変更されたファイル、ステージ済みの内容、現在のブランチ状況などを確認できる。",
  "tags": ["basic", "working-tree"]
}
```

### 12.2 必須項目

- `id`: 一意な識別子
- `command`: 正解となる Git コマンド
- `description`: 読み札に表示する文章
- `explanation`: 回答後に表示する解説

### 12.3 任意項目

- `tags`: 分類用
- `aliases`: 将来、別表現のコマンド許容に使う可能性あり

## 13. 出題内容の方針

初期収録は `30〜40問` 程度を目安とする。  
幅広いユーザーが楽しめるよう、特定の高度運用だけに偏らず、日常的に遭遇しやすいコマンドを中心に構成する。

収録候補カテゴリ:

- 状態確認
- ステージング
- コミット
- 履歴確認
- ブランチ操作
- 取り消し / 戻し
- リモート操作
- 一時退避
- 差分確認
- タグ

収録候補コマンド例:

- `git status`
- `git add`
- `git commit`
- `git push`
- `git pull`
- `git fetch`
- `git switch`
- `git checkout`
- `git branch`
- `git merge`
- `git rebase`
- `git stash`
- `git stash pop`
- `git restore`
- `git restore --staged`
- `git reset`
- `git log`
- `git diff`
- `git remote -v`
- `git tag`

## 14. 出題ロジック

- 開始時に問題数ぶんの問題をランダム抽出する
- 1 プレイ中に同じ問題は重複させない
- 各問題の選択肢には正解 1 件と不正解複数件を含める
- 不正解候補は、正解とカテゴリが近いものを優先できると望ましい
- ただし MVP ではランダム抽出でよい

## 15. 状態管理

SolidJS のシグナルで以下を管理する。

- 現在の画面
- 選択された難易度
- 選択された問題数
- 出題済み問題リスト
- 現在の問題インデックス
- 現在の選択肢
- 回答済み状態
- 正解数
- 回答結果表示状態

## 16. アクセシビリティ

- ボタンと取り札はキーボード操作可能にする
- コントラスト比を意識する
- `prefers-reduced-motion` に対応する
- 装飾目的の背景表現は可読性を阻害しない
- 正誤は色だけでなく文言でも伝える

## 18. CI/CD 要件

### 18.1 品質チェック

GitHub Actions で以下を実行する。

- `ESLint`
- `Prettier` のチェック
- `build`

実行タイミング:

- `push`
- `pull_request`

### 18.2 デプロイ

`main` ブランチへの push 時に以下を実行する。

1. 依存関係をインストール
2. アプリをビルド
3. 生成された静的ファイルを `gh-pages` ブランチへデプロイ

### 18.3 GitHub Pages 前提

- Vite の `base` 設定は GitHub Pages 配備を考慮する
- リポジトリ名 `fudagit-web` 配下で動作できるようにする

## 19. 完了条件

以下を満たしたら MVP 完了とする。

- スタート画面で難易度と問題数を選べる
- 読み札の説明文が少しずつ浮かび上がる
- 取り札を選んで回答できる
- 毎問、正誤と正解と解説が表示される
- 最終結果が表示される
- PC とモバイルでレイアウト破綻がない
- lint / format / build が通る
- `main` push で GitHub Pages に自動デプロイできる
